# Reflection 维度对齐基线

最后更新：`2026-05-01`

## 理论背景

- 原始理论 PDF：`docs/theory/精简-如何实现幸福.pdf`
- 本文用于把理论里的“思考日志”翻译成当前产品和 AI 访谈要达成的技术定义。
- 截至 `2026-05-01`，当前代码已经按本文完成 reflection 的理论对齐、提问策略、完成标准、正文成稿、质量门、fallback、标题治理和用户边界收束。

## 1. 理论目标

- `reflection / 思考` 不是单纯记录“今天想了什么”。
- 它真正要做的是：从当天具体片段里看见新的规律、方向、优势、盲点或判断依据。
- 这类日志的长期价值，是把一天的经历转成可回看的判断资产。
- 它的深度只到“增强判断”，不能写成行动计划、心理诊断、人生结论或稳定公式。
- 如果用户重点变成“下次怎么做”，应转向 `improvement`；如果核心是感谢、支持或被善待，应转向 `gratitude`。

## 2. 最终日志状态

一篇合格的 reflection 日志，读完后应该让人知道：

- 今天哪一个具体片段触发了思考。
- 原来模糊、误判或没看清的地方是什么。
- 今天从这个片段里看见了什么新理解。
- 这个新理解为什么能从当天材料里自然推出。
- 如果材料足够，读完后还能看到：以后判断类似事情时，多了一条什么线索。

这篇日志的核心，不是“我想了很多”，而是“我更清楚自己以后可以凭什么判断”。

## 3. 必填槽位

当前 API 仍沿用 legacy 字段，但 reflection 的内部语义固定如下：

- `trigger = event`
  - 触发思考的具体片段。
  - 不能只是“今天想了很多”“我有点焦虑”。
- `insight = whyItMattered`
  - 从片段里看见的新规律、新理解、优势、盲点或判断依据。
  - 不能是空泛人生感悟。
- `viewpointShift = selfPattern`
  - 视角变化或判断线索。
  - 它回答的是：原来怎么看，现在更清楚了什么；以后判断类似事情时，多了一条什么依据。

## 4. 可选槽位

- `feeling = feeling`
  - 当时的警醒、疑惑、松动、笃定等感受。
  - 它能帮助正文保留现场感，但不能代替新理解。
- `reflectionType = happinessType`
  - 当前固定为三类：
    - `规律发现型`
    - `方向优势型`
    - `判断校准型`

## 5. 完整完成标准

完整模式成立需要：

- 已找到可信 `trigger`
- 已说清可信 `insight`
- 已形成可信 `viewpointShift`

完整模式下，结尾允许轻轻收束到判断线索，例如：

- 以后判断进展时，要看判断依据有没有变清楚。
- 类似事情里，我不能只看忙不忙，而要看真正推进了什么。
- 这让我更清楚，优势不是做得多，而是能把关键判断说清。

完整模式仍然禁止：

- 写成行动计划
- 写成心理诊断
- 写成人生答案
- 写成“只要……就一定……”的稳定公式

## 6. 部分完成标准

如果用户明确不想继续深挖，只要满足下面条件，就允许生成“当前版本日志”：

- 已找到可信 `trigger`
- 已说清可信 `insight`

部分模式下允许写：

- 这个片段带来的当前理解
- 为什么这个理解有证据感
- 现在多了一点判断依据

部分模式下明确禁止：

- 硬写 `viewpointShift`
- 假装已经形成稳定判断标准
- 把当前理解写成长期人生规律

如果用户拒绝继续，但没有具体片段或只有情绪没有新理解：

- 不能继续硬追问。
- 不能硬生成日志。
- 应进入低压选择：“只补一句 / 换一个片段 / 先退出”。

## 7. 写作控制层

reflection 的正文生成沿用统一写作控制接口：

- `voiceMode: "journal"`
  - 像用户自己整理出来的日志，不像系统分析报告。
- `narrativeOrder: "scene_core_shift_close"`
  - 正文顺序默认是：
    - 从触发思考的具体片段进入
    - 写清原来的疑问或判断
    - 写清当天看到的证据和新理解
    - 最后轻收视角变化或判断线索
- `closingMode`
  - `complete -> "stable_clue"`：轻收判断线索
  - `partial -> "current_understanding"`：只停在当前理解
- `toneBanSet`
  - 人生感悟
  - 心理诊断
  - 行动计划
  - 方法论总结
  - 复盘报告
  - 建议腔

## 8. 质量门规则

reflection 至少要拒收以下问题：

- `missing_scene_anchor`
  - 全文没有可信触发片段。
- `missing_reflection_insight`
  - 没有新理解，只是在复述事件或情绪。
- `action_plan_tone`
  - 写成“以后要怎么做”的改进计划。
- `diagnosis_tone`
  - 写成心理诊断或人格判断。
- `life_conclusion_tone`
  - 从一次片段跳到人生答案或宏大结论。
- `partial_fake_judgment_clue`
  - partial 模式下硬写稳定判断线索。

## 9. fallback 原则

reflection 的 fallback 也必须像日志，不像说明文。

正文骨架固定为：

- 触发片段起笔
- 写出新理解
- 写出当时感受或思考类型
- 完整模式轻收判断线索；部分模式停在当前理解

完整模式 fallback：

- 可以自然写出可信 `viewpointShift`
- 结尾力度只到“以后判断类似事情时多了一条线索”

部分模式 fallback：

- 只能停在“这件事让我多了一层判断依据”
- 不能假装已经形成稳定标准
