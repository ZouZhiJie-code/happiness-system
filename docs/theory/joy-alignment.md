# Joy 维度对齐基线

最后更新：`2026-05-01`

## 理论背景

- 原始理论 PDF：`docs/theory/精简-如何实现幸福.pdf`
- 本文用于把理论里的“开心日志”翻译成当前产品和 AI 访谈要达成的技术定义。

## 1. 理论对齐

### 已经做到的

- `joy` 已经不再只是普通流水账，而是围绕真实发生过的片段做结构化追问。
- 系统已经能从很小的开心片段开始，而不是要求用户一开始就说出“大事”。
- joy 已经有专属槽位，不再只复用 `event / reason / type` 这种旧复盘结构。
- joy 现在已经区分两条完整路径：
  - `meaning_track`：沉淀出可信 `manualClue`
  - `delight_track`：沉淀出可信 `delightSignature`
- 连续几轮找不到可信开心片段时，系统已经会建议转去 `improvement`。
- 用户点击“生成日志”后，系统当前会把结构化信息压到内部，只把日志正文给用户看。

### 还没做到的

- joy 正文虽然已经从结构卡转向正文优先，但还没有完全达到产品级成稿质感。
- “兴趣 / 热爱 / 未来方向信号”已经有字段，但还没形成足够强的产品表达和长期利用规则。
- “对外界有正向影响的开心含金量更高”已有 `valueImpact` 字段，但还没有真正升级成更强的规则系统。
- 跨天长期汇总规则还没有做，不能把多篇 joy 日志继续沉淀成稳定的长期使用说明书。

### 仍然做偏的地方

- 当前正文已经更自然，但某些 fallback 文本仍然偏“解释访谈结果”，不像用户自己写下来的日志。
- joy 已经实现了 `manualClue`，但“理论里的生命力线索”在产品文字层还可以更克制、更贴近用户口吻。
- `improvement` 已在 `2026-05-01` 完成理论规格、结构字段扩展、AI 抽取独立化、fallback 抽取、专属提问策略、完成收束、正文生成、质量门、fallback draft、标题治理和自动化验收样例；`gratitude` 仍主要是通用壳子，还不能算真正对齐。`fulfillment` 已在 `2026-04-30` 完成单独理论对齐，`reflection` 已在 `2026-05-01` 完成单独理论对齐。

### 必须继续保持为规则的地方

- `meaning_track` 的 `manualClue` 与 `delight_track` 的 `delightSignature` 都必须继续由规则判断，不能退回只靠 prompt 判断。
- 如果用户明确表示不想继续提炼规律，系统可以放行生成“当前版本日志”，但这条例外必须由服务层规则判断，不能让模型自由决定。
- 如果用户明确表示不想继续追问，但材料不足以生成当前版本日志，系统必须尊重边界，给低压选择，而不是继续补槽位式追问。
- 连续找不到可信开心片段时，必须继续由服务层触发 `dimension_redirect`，不能让模型临场发挥。
- 哪些字段算“可信写入”，必须继续靠结构化抽取与证据门槛，而不是关键词命中就硬写。
- 用户不可见 `snapshotData`，这条边界必须保持；结构化线索只能作为系统内部状态存在。

## 2. 结构设计

### 最终理想日志状态

一篇合格的 joy 日志，应该让人读完后知道：
- 哪个片段让我开心。
- 真正让我有感觉的点是什么。
- 它把我带进了什么更好的状态，或满足了我什么在乎 / 需要。
- 这说明什么环境、关系、事情更容易让我进入好状态。
- 它是否顺手露出了兴趣、方向或价值感的信号。

### 必须挖出的槽位

- `joyMoment`
- `joySource`
- `stateShift`
- `meaningNeed`
- `manualClue`
- `delightSignature`

### 可选槽位

- `directionSignal`
- `valueImpact`
- `durability`
- `tags`

### 完成标准

- 已经找到一个可信的 `joyMoment`
- 已经说清 `joySource`
- `meaning_track`：已经拿到 `stateShift` 或 `meaningNeed` 至少一个，并沉淀出可信 `manualClue`
- `delight_track`：已经拿到可信 `stateShift`，并沉淀出可信 `delightSignature`
- 例外情况下：用户明确拒绝继续提炼规律时，可在前 3 项成立后生成当前版本日志
- 如果连续几轮仍找不到可信片段，不应硬完成，而应建议转 `improvement`
- 如果用户拒绝继续且核心材料不足，不应硬生成，而应提供“只补一句 / 换一个片段 / 先退出”

### 影响进度和收尾的槽位

- 强影响进度：`joyMoment`、`joySource`
- 中段推进：`stateShift`、`meaningNeed`
- 默认决定是否可收尾：
  - `meaning_track -> manualClue`
  - `delight_track -> delightSignature`
- 例外放行条件：用户明确拒绝继续提炼规律
- 边界不足条件：用户明确拒绝继续追问但核心材料不足
- 只决定是否值得补问一轮：`directionSignal`、`valueImpact`、`durability`

## 3. 访谈策略

### 应该分几步聊

- 第 1 步：找片段
- 第 2 步：找真正开心点
- 第 3 步：找状态变化或被满足的在乎
- 第 4 步：沉淀成使用说明书线索
- 可选补充：补问方向信号或价值影响

### 每一步的目标

- 第 1 步：找到真实、具体、值得展开的开心瞬间
- 第 2 步：确认到底什么让用户开心，不停在事件表面
- 第 3 步：确认这份开心为什么有分量，落到状态变化或在乎 / 需要
- 第 4 步：把前面的信息压缩成可复用规律

### 最容易跑偏的地方

- 停在“开心、轻松、温暖”这类情绪标签，没有进入真正开心点
- 过早贴类型标签，代替了实际理解
- 最后把规律提炼成正确但没用的空话
- 正文生成时把内部结构写得太明显，像总结卡而不像日志

### 什么时候继续深挖

- 已经有事件，但还没有真正开心点
- 已经有开心点，但还没有状态变化或在乎 / 需要
- 已经有状态或在乎，但还没有 `manualClue`
- 已经出现明显方向或价值信号，值得多问一轮

### 什么时候停止

- 已经有 `joyMoment`
- 已经有 `joySource`
- 已经有 `stateShift / meaningNeed`
- 已经有 `manualClue`
- 或者用户明确表示不想继续提炼规律，且前 3 项已经成立
- 额外信号不明显时，不继续硬拔高
- 连续几轮都找不到可信片段时，停止 joy 主线，建议转 `improvement`

## 4. 当前产品补充说明

- 结构化线索现在只保留在系统内部，用来驱动进度、收尾和日志生成。
- 用户最终看到的是“日志正文”，不是 joy 结构卡。
- 因此 joy 的下一步重点已经不是继续暴露结构，而是把正文写得更像一篇真正属于用户的开心日志。

## 5. Joy 正文成稿规格（已锁定，已进入当前实现基线）

这一节只定义 joy 正文之后必须达到的产品标准。

当前结论是：
- 这是 `2026-04-29` 锁定的成稿规格。
- 这一节不是 prompt 草案，也不是技术实现细节。
- 当前 joy 的 prompt、质量门和 fallback 已经按这套规格联动。
- 后续其他维度要复用这套方法时，以 `docs/theory/dimension-draft-template.md` 为模板入口。

### 读感定位

- 最终成稿要介于“私密日记”和“复盘记录”之间。
- 用户读起来应该觉得“这是我整理过后的表达”，而不是“系统帮我分析了一次”。
- 可以有整理后的清晰度，但不能强到像总结卡、汇报稿或访谈结论。

### 标题规格

- 标题要像日志标题，不像系统字段。
- 标题优先取具体片段、场景、关系或感受锚点。
- 不使用“开心日志”“本次开心：”这类系统前缀。
- 当前标题上限固定为 `16` 字，保持单行可读，不依赖横向滑动查看。
- 标题默认应短、具体、方便回看。
- 标题必须经过后端语义短标题治理；AI 返回的流水句、字段句、截断句不能直接给用户。
- 标题不能通过把长事件句 `slice(0, 16)` 得到。

### 正文核心内容

- 正文必须落到具体片段，不能从抽象判断起笔。
- 正文必须写出真正让用户开心的点，而不只是复述发生了什么。
- 正文必须自然带出状态变化或被满足的在乎 / 需要，至少一个要成立。
- 如果对话里已经出现方向感、价值感、持续性信号，可以自然写入，但前提是证据足够，不能硬拔高。

### 正文组织方式

- 如果只有一个片段，就围绕这个片段写完整。
- 如果有多个片段，采用“并列记录”的组织方式。
- 并列记录不等于平均铺开；更有分量的片段可以占更多篇幅。
- 最终读感仍然要像一篇日志，而不是几个模块并排罗列。
- 不预设固定段数、固定句数，长度跟随材料密度走，能总结多少就总结多少，实事求是。

### 语气与口吻

- 默认使用第一人称。
- 语气克制、真诚、贴近用户自己的表达。
- 不鸡汤，不夸张，不空泛抒情。
- 不做建议，不解释产品，不解释访谈过程，不替用户上价值。
- 可以有回看和提炼感，但不能有“讲道理”的分析腔。

### 完整模式的结尾

- 如果这轮已经形成可信 `manualClue`，结尾要轻轻收束出这条线索。
- 这条线索必须像自然体会，不像方法总结，不像公式，不像操作说明。
- 读完后应该让人隐约感觉“我更知道自己在什么情况下会进入好状态了”，但不能写得太满。

### 部分模式的结尾

- 如果这轮只是“当前版本日志”，结尾只能停在当前发现。
- 可以写“我现在更知道自己会被什么打动”或“我更在意什么”。
- 不能假装已经形成稳定规律。
- 部分模式下，禁止写出强结论式、可复用式、条件句式的规律表达。

### 明确禁止的写法

- 禁止字段感写法：不能出现槽位词、总结卡口吻、分点式表达。
- 禁止系统感写法：不能出现“这次访谈”“我已经整理出”“使用说明书”“当前版本日志”这类元叙述。
- 禁止建议腔：不能出现“以后可以”“我应该”“这提醒我要”这类收尾。
- 禁止假规律：尤其在部分模式下，不能写成“只要……我就更容易……”这种稳定公式。
- 禁止为完整而补写：用户没表达过的情节、关系含义、价值判断，不能脑补。

### 验收标准

- 用户看到正文时，第一反应应是“这是我的一篇日志”，而不是“这是系统对我的分析”。
- 用户回看时，应能重新进入那个开心片段，并知道真正打动自己的点。
- 完整模式下，读完会隐约获得一条关于自己开心来源的线索，但不会觉得被硬总结。
- 部分模式下，读完只会感觉“我现在更明白了一点”，不会误以为自己已经形成稳定规律。
- 多片段场景下，读感应是“一天里几个开心瞬间被自然记下来了”，不是条目清单。
