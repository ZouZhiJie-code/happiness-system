# Gratitude 维度对齐基线

最后更新：`2026-05-01`

## 理论背景

- 原始理论 PDF：`docs/theory/精简-如何实现幸福.pdf`
- 本文用于把理论里的“感谢日志”翻译成当前产品和 AI 访谈要达成的技术定义。
- 截至 `2026-05-01`，gratitude 的目标是从通用壳子升级为完整理论对齐维度。

## 1. 理论目标

- `gratitude / 感谢` 的核心任务，不是礼貌地说谢谢。
- 它真正要确认的是：今天谁看见并回应了我的需要。
- 理论里说，值得感谢的人做了一件关键的事：他们看见了我的需要，并回应了我的需要。
- 因此，感谢日志要训练用户从“只看自己”转向“看见他人”，慢慢培养关系能力和利他意识。
- 长期来看，这会帮助用户建立更稳定、深厚、长久的关系，并形成一种不再孤立的共同体感觉。

AI 在这个维度里最后要帮用户沉淀的，不是“我很感动”，而是一条可信的关系线索：

- 什么样的人值得珍惜。
- 什么样的行为值得学习。
- 我在关系里会因为什么样的回应感到被支持、被理解或被接住。

## 2. 最终日志状态

一篇合格的 gratitude 日志，读完后应该让人知道：

- 今天哪个具体片段让我想表达感谢。
- 我想感谢谁，或感谢哪段关系里的善意。
- 对方具体做了什么。
- 对方看见并回应了我当时的什么需要、难处、压力或在乎。
- 这份回应为什么对我重要。
- 如果材料足够，读完后还能看到：什么样的关系回应值得我珍惜、学习或回馈。

这篇日志的核心，不是“对方人很好”，而是“我看见了关系里一份具体、真实、回应到我的善意”。

## 3. 必填槽位

没有下面这些槽位，就不能认为一篇 gratitude 日志已经完整成立：

- `gratitudeMoment`
  - 真实发生过的感谢片段。
  - 不能只是“我感谢家人”“我感谢朋友”。
- `gratitudeTarget`
  - 感谢对象或关系来源。
  - 可以是具体的人，也可以是家人、同事、朋友、团队等关系对象。
- `kindAction`
  - 对方具体做了什么。
  - gratitude 的核心不是善意标签，而是具体行为。
- `seenNeed`
  - 对方看见并回应了我的什么需要或难处。
  - 这是 gratitude 区别于普通开心、充实和思考的核心槽位。
- `gratitudeReason`
  - 为什么这份感谢重要。
- `relationshipSignal`
  - 完整模式下的高阶槽位。
  - 它回答的是：什么样的关系回应值得我珍惜、学习或回馈。

当前 legacy 投影：

- `event = gratitudeMoment`
- `feeling = innerEffect`
- `whyItMattered = gratitudeReason`
- `happinessType = gratitudeType`
- `selfPattern = relationshipSignal`

## 4. 可选槽位

- `innerEffect`
  - 这份善意带来的内在影响，例如被接住、被理解、安心、松一口气、不再孤立。
- `gratitudeType`
  - 当前按五类收束：
    - `支持回应型`
    - `理解体谅型`
    - `陪伴接住型`
    - `照顾减负型`
    - `信任机会型`
- `reciprocityHint`
  - 用户自然表达的学习、回馈或回应意愿。
  - 不能硬写成“我应该报答”或“我欠了人情”。
- `tags`

## 5. 完整完成标准

完整模式成立需要：

- 已找到可信 `gratitudeMoment`
- 已说清 `kindAction`
- 已说清 `seenNeed`
- 已说清 `gratitudeReason`
- 已形成可信 `relationshipSignal`

完整模式下，结尾允许轻轻收束到关系线索，例如：

- 我更知道，这样能看见我真实需要的人值得珍惜。
- 关系里最打动我的，是这种不催促、但认真接住我的回应。
- 我也想学习这种先理解别人处境的方式。

完整模式仍然禁止：

- 写成感谢信模板。
- 写成报答任务或道德负债。
- 从一次善意直接上升成完美关系判断。
- 写成“做人要懂得感恩”的说教。

## 6. 部分完成标准

如果用户明确不想继续深挖，只要满足下面条件，就允许生成“当前版本日志”：

- 已找到可信 `gratitudeMoment`
- 已说清可信 `kindAction`
- 已说清 `seenNeed` 或 `gratitudeReason` 至少一个

部分模式下允许写：

- 具体感谢片段。
- 对方做了什么。
- 这件事为什么让我想感谢。
- 当前我感到被支持、被理解或被接住的一点。

部分模式下明确禁止：

- 硬写 `relationshipSignal`。
- 假装已经形成稳定关系判断。
- 把当前感谢写成长期人际原则。
- 写成“我应该回报对方”的行动要求。

如果用户拒绝继续，但还没有具体行为或感谢原因：

- 不能继续硬追问。
- 不能硬生成日志。
- 应进入低压选择：“只补一句 / 换一个片段 / 先退出”。

## 7. 写作控制层

gratitude 的正文生成沿用统一写作控制接口：

- `voiceMode: "journal"`
  - 像用户自己整理出来的日志，不像感谢信、表扬稿或系统总结。
- `narrativeOrder: "scene_core_shift_close"`
  - 正文顺序默认是：
    - 从具体感谢片段进入。
    - 写清对方具体做了什么。
    - 写清对方回应了我什么需要。
    - 写清这份感谢为什么重要。
    - 完整模式轻收关系线索；部分模式停在当前感谢。
- `closingMode`
  - `complete -> "stable_clue"`：轻收关系线索。
  - `user_override_partial -> "current_understanding"`：只停在当前感谢。
- `toneBanSet`
  - 感谢信模板。
  - 表扬稿。
  - 道德负债感。
  - 报答任务。
  - 人际建议腔。
  - 鸡汤式善意。

## 8. 质量门规则

gratitude 至少要拒收以下问题：

- `missing_scene_anchor`
  - 全文没有可信具体片段。
- `missing_gratitude_action`
  - 没有写清对方具体做了什么。
- `missing_seen_need`
  - 没有写清这份善意回应了用户什么需要或难处。
- `thank_you_template_tone`
  - 写成感谢信、公文式感谢或表扬稿。
- `moral_debt_tone`
  - 写成欠人情、必须报答、无以为报。
- `advice_tone`
  - 写成做人要感恩、以后必须回报。
- `empty_kindness_tone`
  - 只写“对方很好 / 很善良 / 很温暖”，没有具体行为。
- `partial_fake_relationship_signal`
  - partial 模式下硬写稳定关系线索。

## 9. fallback 原则

gratitude 的 fallback 也必须像日志，不像感谢信。

正文骨架固定为：

- 感谢片段起笔。
- 写清对方具体行为。
- 写清被看见或被回应的需要。
- 写清这份感谢为什么重要。
- 完整模式轻收关系线索；部分模式停在当前感谢。

完整模式 fallback：

- 可以自然写出可信 `relationshipSignal`。
- 结尾力度只到“我更知道什么样的关系回应值得珍惜或学习”。

部分模式 fallback：

- 只能停在“这件事让我感到被回应过”。
- 不能假装已经形成稳定关系判断。
- 不能硬写回馈任务。

## 10. 标题治理与用户边界

标题要求：

- 标题上限保持 `16` 字。
- 标题优先从 `seenNeed / kindAction / relationshipSignal / gratitudeReason` 提炼。
- 典型方向包括 `被稳稳接住 / 被认真理解 / 那句及时提醒 / 有人帮我理清 / 被信任的机会`。
- 不能使用 `感谢日志 / 今天的感谢` 作为有材料时的默认标题。
- 不能把长事件句机械截断成标题。

用户边界要求：

- 用户明确表示“不想继续 / 不要再追问 / 直接生成 / 总结日志 / 整理成日志 / 追问没有意义”时，边界优先级高于槽位完整度。
- 材料足够时允许 partial 当前版本日志。
- 材料不足时不能继续补槽位式追问，应提供“只补一句 / 换一个片段 / 先退出”。
