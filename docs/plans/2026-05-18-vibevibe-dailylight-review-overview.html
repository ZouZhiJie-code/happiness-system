Daily Light 体系化评审总览

评审口径按你定义的版本执行：以 Daily Light 自身产品路线为主轴，覆盖 本地稳定开发 / 小范围真实用户可用 / 可正式上线运营 三层目标；只有仓库里已经有 代码、配置、验证链路 才算完成。外部参考来自 vibevibe 的 8 个高级章节，但不会机械按教程补作业。

日期：2026-05-18 分支：main 远端：git@github.com:ZouZhiJie-code/happiness-system.git 判断口径：代码闭环优先 结论力度：强决策版

一句话结论

本地开发：偏强

产品主链、鉴权、日历、分析、画像等主要业务面都有真实代码和大量单测覆盖，`typecheck` 当前可过。

小范围试用：可做，但要补两刀

账号体系与私有页守卫已经成立，但默认测试入口不稳定、部署配置尚未实体化，试用前需要补测试边界和最小发布面。

正式上线：还不能算完成

域名、CI/CD、Serverless/VPS 部署、监控、日志、备份、证书、生产环境配置目前缺少仓库内实物闭环。

总判断

能力面 当前判断 为什么 决策
后端 API 已完成到产品主链可用 仓库里存在 28 个业务/API 路由，calendar / analysis / interview / draft / profile / auth / happiness-score / daily-journal 都有服务层和测试。 现在必须保住，不用重做架构。
认证与安全基础 已完成到首版可用 `User`、`AuthSession`、密码 hash、`httpOnly` cookie、私有页守卫、删号级联、注册协议链路都真实存在。 现在必须保住，后续只补生产安全项。
测试与自动化 部分完成 主仓有 71 个 tracked test 文件，定向排除 `.worktrees/**` 后核心 API/鉴权测试可过；但 `vitest.config.ts` 本身没有写 exclude，`npm test` 默认入口会被历史 worktree 噪声污染。 现在必须修到默认入口可信。
Git 协作 部分完成 远端仓库、主分支、worktree 习惯、上线文档都存在；但仓库里没有 PR/CI 配置实体，更多是流程意识而不是自动门禁。 现在必须补最小协作门槛，避免“文档有、机器不执行”。
公网访问 / 临时分享 认知有，实物弱 仓库里没有 tunnel、preview、公开访问配置；更接近“知道该怎么做”，还不是“仓库已具备”。 可后置到试用前一批完成。
Serverless 部署 / CI-CD 未完成 未发现 `vercel.json`、`.github/workflows/*`、平台部署配置、preview/production 分环境脚手架。 正式上线前必须补。
域名 / DNS / HTTPS 未完成 仓库内没有域名绑定、DNS、证书、入口统一、备案/地域策略等可执行配置或操作记录。 正式上线前必须补。
VPS / 运维 现阶段不建议投入 仓库没有 Dockerfile、compose、Nginx/OpenResty、1Panel、备份与监控配置；对当前单产品阶段，先上托管平台更符合性价比。 现阶段不建议先走 VPS 主线。

现在必须做

1. 修默认测试入口

  • `vitest.config.ts` 只有 `environment / globals / setupFiles`，没有任何 `include/exclude`。
  • README 明写“默认排除 `.worktrees/**`”，但真实配置没有落进去，导致 `npm test` 与文档承诺不一致。
  • 这会直接削弱“改完代码能不能信”的判断基础。

2. 固化一条真实部署主线

  • 先选托管平台路线，不要同时开 Serverless 和 VPS 两条线。
  • 把 build 命令、环境变量、preview / production 区分、最小 smoke check 固化到仓库。
  • 否则试用只能靠“本机能跑”和口头步骤维持。

3. 补生产环境变量与发布清单

  • 数据库、AI provider endpoint、APP URL、cookie 安全项都要形成正式部署清单。
  • 现在有 `.env` 认知，但没有生产部署配置闭环。

4. 把语音转写继续留在关闭态

  • `/api/transcribe` 当前明确是 stub，只返回占位 transcript。
  • 它不在当前主链里，可以继续隐藏；一旦开放给真实用户，就必须先接真实模型。

可以后置

公网临时分享

如果近期只是小范围验证，可以先用 preview URL 或 tunnel 做演示，但不要把这当正式交付面。

PR 自动门禁

协作自动门禁可以和部署主线一起补，不必先追完整 GitHub Flow 套餐。

完整监控与成本面板

正式上线前要有，试用期可以先从平台日志、错误统计和简单健康检查开始。

画像 / 记忆增强

这块已有实现，但 `memoryEnabled` 默认关闭；它不是上线阻断项，应排在主链稳定之后。

现阶段不建议投入

先上 VPS 再说

教程第 14 章要求的是完整运维责任。对 Daily Light 当前阶段，先走托管平台更符合资源效率，也更贴近现在缺的是“发布闭环”而不是“服务器控制权”。

追求测试全覆盖

现阶段问题不是测试数量不够，而是默认入口不可靠、主仓与历史 worktree 边界没封好。优先修测试边界，再谈扩面。
