公网访问 / Serverless / CI-CD 评审

外部章节基线：10 localhost public access、12 serverless deploy cicd

对 Daily Light 的结论：这组目前更多停留在认知层和计划层，还没有形成仓库内的部署实体。

证据

  • 仓库里没有发现 vercel.json、没有部署平台配置、没有 `.github/workflows/*`、没有 preview / production 自动化配置。
  • README 和 launch 文档说明了“如何本地跑、如何手动验收、如何判断上线”，但没有把平台部署主线固化成机器可执行资产。
  • `/api/transcribe` 当前仍是 stub，这说明“开放哪些能力进公网环境”本身还处在收口阶段。
  • 小范围可用所需的最小动作，如平台环境变量、preview 链接、部署日志入口、生产 smoke check，在仓库里都没有实体化痕迹。

为什么这组不能按完成算

教程能力 Daily Light 当前状态 判断
本地到公网的过渡意识 有，团队显然理解本地运行、launch 验收和试用的区别。 部分完成
正式部署主线 未看到平台配置实体。 未完成
preview / production 分离 文档有上线计划，但没有部署环境配置。 未完成
自动化部署 / webhook / CI 未看到任何实际工作流文件。 未完成
部署日志与成本面板 暂无仓库证据。 未完成

强决策

  • 现在必须做：选定一条托管平台主线，把 build 命令、环境变量、preview / production、最小 smoke test 固化到仓库。
  • 可以后置：复杂的成本优化、灰度矩阵、跨平台多云冗余。
  • 现阶段不建议：一边没有平台主线，一边先去搭 VPS。当前应该先让“每次提交都能稳定发布到一个正式可访问环境”。
