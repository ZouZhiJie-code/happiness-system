# 设置页重构设计文档

## 1. 概述

将设置页从单一访谈偏好页面扩展为账户级全局设置页面，包含5个核心模块，支持左右分栏交互，延续暖棕色木纹风格但采用简约设计语言。

## 2. 功能模块

### 2.1 个人资料
- 头像（URL或上传）
- 昵称
- 个人简介（可选）

### 2.2 访谈偏好（已有，迁移）
- 记忆能力开关（memoryEnabled）
- 转写失败自动回退（transcriptAutoFallbackEnabled）
- 追问深度（温和/适中/深入）
- 引导语开关

### 2.3 通知偏好
- 邮件提醒开关
- 系统通知开关

### 2.4 提醒设置
- 访谈提醒（每日/每周定时提醒，可设置时间）
- 月度回顾提醒

### 2.5 数据管理
- 导出日志（JSON格式）
- 清除历史数据（需二次确认）
- 数据存储说明

## 3. 交互设计

### 3.1 布局
- **左侧导航**：固定宽度（240px），显示5个模块名称和图标，当前选中项高亮（左侧2px色条）
- **右侧内容区**：根据选中模块动态切换，内容直接平铺，用留白和分割线区分区块
- **响应式**：移动端改为上下堆叠，左侧导航改为顶部水平滚动标签

### 3.2 表单交互
- 模块切换：点击左侧导航项，右侧内容平滑切换（无页面刷新）
- 表单保存：每个模块独立保存，保存按钮固定在内容区底部
- 表单验证：使用 Zod schema 即时验证，错误信息就近显示
- 状态反馈：保存成功/失败的 toast 提示

### 3.3 组件风格（简约设计）
- 去掉模块内容区的独立卡片包裹
- 表单项用底部细线分隔，不用厚重卡片
- Toggle Switch：无边框，浅灰色轨道，选中时暖棕色
- 输入框：只有底部边框，聚焦时底部变色
- 下拉菜单：简约下拉，无多余装饰

## 4. 视觉设计

### 4.1 颜色系统
- 背景：`rgba(255,249,239,0.34)` 浅暖色
- 边框：`rgba(115,77,39,0.14)` 暖棕
- 文字：`#2f2217`（标题）、`#5a4632`（正文）、`#6a5e53`（辅助）
- 强调色：`#d89d59`（金色）、`#8ba17a`（绿色）

### 4.2 左侧导航
- 背景色与内容区区分（略深）
- 选中项：左侧 2px 色条强调 + 文字加粗
- 图标：简单线条图标，与现有风格统一

### 4.3 模块内容区
- 标题 + 描述文字 + 表单项，垂直排列
- 表单项之间用 1px 分割线或间距区分
- 保存按钮固定在内容区底部，sticky 定位

## 5. 数据层设计

### 5.1 数据库模型

```prisma
model UserSettings {
  id            String   @id @default(cuid())
  userId        String   @unique
  // 个人资料
  nickname      String?
  avatar        String?
  bio           String?
  // 访谈偏好
  interview     Json     @default("{}")  // { memoryEnabled, transcriptAutoFallback, interviewStyle, guideEnabled }
  // 通知偏好
  notification  Json     @default("{}")  // { emailEnabled, systemEnabled }
  // 提醒设置
  reminder      Json     @default("{}")  // { dailyReminder, weeklyReminder, monthlyReview }
  // 数据管理
  dataManagement Json    @default("{}")  // { exportFormat }
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt
}
```

### 5.2 API路由

- `GET /api/settings` - 获取用户设置
- `PUT /api/settings` - 更新用户设置（支持部分更新）
- `POST /api/settings/export` - 导出日志数据
- `DELETE /api/settings/data` - 清除历史数据（需二次确认）

### 5.3 前端状态管理

- 使用 React Query 管理设置数据
- 乐观更新：保存时立即更新 UI，失败时回滚
- 缓存策略：设置数据长期缓存，手动失效

### 5.4 数据流

1. 页面加载 → `GET /api/settings` → 缓存到 React Query
2. 用户修改 → 本地表单状态
3. 点击保存 → `PUT /api/settings` → 成功后更新缓存
4. 导出/清除 → 调用专用 API → 下载文件/确认清除

## 6. 实现优先级与计划

### 阶段一：基础框架（1-2天）
1. 创建 `UserSettings` 数据库模型和迁移
2. 实现 `GET/PUT /api/settings` API
3. 搭建设置页左右分栏布局框架
4. 实现个人资料模块（头像、昵称、简介）

### 阶段二：访谈偏好迁移（1天）
1. 将现有访谈偏好迁移到新数据模型
2. 在新布局中实现访谈偏好模块
3. 确保与现有功能兼容

### 阶段三：通知与提醒（1-2天）
1. 实现通知偏好模块
2. 实现提醒设置模块
3. 对接提醒功能（可后续迭代）

### 阶段四：数据管理（1天）
1. 实现日志导出功能（JSON格式）
2. 实现数据清除功能（带二次确认）
3. 数据存储说明文案

### 阶段五：优化与测试（1天）
1. 响应式适配
2. 动画与过渡效果
3. 错误处理与边界情况
4. 编写测试

## 7. 并行开发注意事项

### 7.1 依赖关系
- **数据库迁移** → **后端API** → **前端页面**（串行依赖）
- **前端布局框架** 可以与 **数据库/API** 并行开发（使用mock数据）
- **各模块前端组件** 可以并行开发

### 7.2 并行策略
1. **阶段一并行**：
   - 开发者A：数据库迁移 + API实现
   - 开发者B：前端布局框架 + 个人资料模块UI

2. **阶段二并行**：
   - 开发者A：访谈偏好迁移 + 数据兼容性验证
   - 开发者B：通知偏好 + 提醒设置模块UI

3. **阶段三并行**：
   - 开发者A：数据管理API + 导出/清除功能
   - 开发者B：响应式适配 + 动画效果

### 7.3 接口契约
- 前后端并行开发时，先定义好API接口规范（请求/响应格式）
- 使用TypeScript类型定义共享接口契约
- 前端使用mock数据开发，后端API就绪后切换

### 7.4 测试策略
- 单元测试：每个模块独立测试
- 集成测试：设置保存/读取完整流程
- E2E测试：关键用户路径（修改设置→保存→刷新验证）

## 8. 风险与缓解

### 8.1 数据迁移风险
- **风险**：现有访谈偏好数据迁移失败
- **缓解**：迁移脚本需支持回滚，迁移前备份数据

### 8.2 性能风险
- **风险**：设置数据过大影响加载速度
- **缓解**：JSON字段按需加载，使用懒加载策略

### 8.3 兼容性风险
- **风险**：新设置与现有功能冲突
- **缓解**：渐进式迁移，保留旧字段兼容期
