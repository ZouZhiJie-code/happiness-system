测试自动化与 Git 协作评审

外部章节基线：09 testing automation、11 git collaboration

对 Daily Light 的结论：有测试意识，也有大量测试资产，但默认入口和机器门禁还没真正收口。

高信号事实

  • 仓库当前 tracked 测试文件数为 71，说明测试资产并不少。
  • `npm run typecheck` 当前可过。
  • 显式执行 npx vitest run --exclude '.worktrees/**' --exclude '.claude/worktrees/**' ... 后，主仓这组核心 API / 鉴权 / calendar / analysis / draft 测试可通过。
  • 但直接跑 npm test 会把历史 `.worktrees/*` 测试一起扫进来，出现大量无关失败，说明默认入口不可靠。
  • `README.md` 明写“Vitest 当前默认只扫描 `tests/**/*.test.{ts,tsx}` 并排除 `.worktrees/**`”，但真实 `vitest.config.ts` 里没有任何 `include/exclude` 配置。
  • Git 侧已有远端 `origin` 和 `main` 主分支，也有 worktree 与上线文档习惯，但仓库里没有 `.github/workflows/*`、没有 CI 配置实体。

对产品路线的意义

  • 这不是“测试少”的问题，核心问题是默认入口不可信。当团队说“`npm test` 绿了”，现在这句话并不稳定成立。
  • 这不是“不会用 Git”的问题，核心问题是协作门槛没有机器化。现在更多依赖自觉和文档，而不是仓库自动执行。
  • 因此教程第 9、11 章对 Daily Light 的最大启发，不是再补几十个测试，而是先把“主仓边界、默认命令、远端门禁”锁实。

强决策

  • 现在必须做：修 `vitest.config.ts`，把 README 承诺的 `include/exclude` 真正落进去；同时补最小 CI，至少跑 `typecheck + test`。
  • 可以后置：E2E 大规模扩面、复杂 preview matrix、测试覆盖率 KPI。
  • 现阶段不建议：一上来就追求重流程 GitFlow。先把 `main + feature branch + PR + CI` 这条最短主线跑顺。
