# 上线前推进总计划

> **For Codex / Worktree / Subagent:** 本项目后续推进以仓库文档为唯一行动依据。任何实现、验收、修复、回归与 handoff，都先对齐本文件及其关联文档。

关联文档：
- [上线前手动验收矩阵](./2026-05-17-launch-acceptance-matrix.md)
- [上线前问题池](./2026-05-17-launch-issue-tracker.md)

## 目标

把当前 `Happiness-system-codex` 从“功能已基本成形”推进到“可做一轮真实上线判断”的状态。

这里的上线判断标准分两层：
- 产品层：已上线范围内的主要功能尽量全绿。
- 风险层：不存在 `无法注册登录`、`主流程中断`、`丢内容`、`串账号`、`串日期`、`保存结果不可信` 这类阻断问题。

## 主原则

- 仓库是唯一真相源。
- 验收按模块分批执行，不做一镜到底式全站漫游。
- 排序按风险优先，不按页面顺序。
- 任何发现项先入问题池，再决定修复批次。
- 任何修复完成后必须回原用例回归。

## 批次顺序

### 批次 1：账户与数据安全

目标：
- 确认账号体系可用
- 确认登录保护有效
- 确认多账号不串线
- 确认删号级联成立

### 批次 2：核心记录主链路

目标：
- 确认五维访谈可用
- 确认单篇日志生成、编辑、保存成立
- 确认当天完整日志生成、编辑、保存成立
- 确认每个维度的 AI 访谈效果达到当前理论规格，而不只是“功能能跑通”

当前拆分：
- `批次 2A`：功能主链路闭环
- `批次 2B`：AI 访谈效果验收

### 批次 3：状态恢复与工作区切换

目标：
- 确认刷新恢复正确
- 确认 `entryDate` 不串线
- 确认访谈、单篇日志、完整日志切换不丢内容

### 批次 4：异常、空态与上线阻断项

目标：
- 集中识别用户无法继续的错误
- 识别结构化错误反馈是否完整
- 识别 stub / 占位能力的上线处理方式

### 批次 5：Calendar 回看链路

目标：
- 确认 month / week / day 正确表达同一天状态
- 确认 deep link 返回正确日期与维度

### 批次 6：Analysis 与评分回流

目标：
- 确认 overview / score / rhythm / insights 正确依赖真实数据
- 确认当天评分录入后能够正确回流到 analysis

### 批次 7：Profile / Memory / 设置补充能力

目标：
- 确认画像、记忆库、演变、设置这些补充能力不拖主流程后腿

## 优先级定义

### P0

阻断上线。

典型情形：
- 无法注册、登录、退出
- 访谈无法开始或无法继续
- 日志无法保存
- 刷新或切页后内容丢失
- 多账号或跨日期串数据
- analysis / calendar 明显伪造或错投影关键状态

### P1

强烈建议上线前修复。

典型情形：
- 主要链路可以完成，但反馈错误、流程卡顿、状态表达不稳定
- 某些关键空态或错误文案不足，用户会明显迷失
- 某模块结果可信度不足，但不至于完全阻断主流程

### P2

可延后。

典型情形：
- 低频体验问题
- 非关键视觉/文案不一致
- 可明确记录为已知限制的问题

## 协作规则

- 我负责维护这三份主文档的一致性。
- `worktree` 和 `subagent` 执行前必须先读相关文档。
- 计划变更先改仓库文档，再开始实现或回归。
- 不通过口头记忆推进，不以飞书或聊天记录替代仓库文档。

## 执行与回归规则

每批执行后固定输出三类结果：
- 通过项
- 未通过项
- 问题清单

每个问题都按以下流程处理：
1. 写入问题池。
2. 关联用例 ID。
3. 标 `P0 / P1 / P2`。
4. 进入修复批次。
5. 修复后回到原用例回归。
6. 在问题池写回归结论。

## 并行与串行策略

### 总原则

- 能并行的优先并行，但只并行“互不争抢数据前置、账号状态、同一文档写入面”的工作。
- 一旦多个任务共享同一个测试账号、同一天来源日志、同一篇完整日志或同一份计划文档，默认转串行。
- 功能链路验收和 AI 访谈质量验收分开看待；前者先确认“能跑”，后者再确认“问得对、收得住、生成可信”。

### 适合并行的块

- 文档标准抽取：
  - 从 `docs/operator-runbook.md`
  - `docs/integration-guide.md`
  - `docs/theory/*-alignment.md`
  - `tests/unit/*`
  - 提取现有验收标准、自动化覆盖和高风险空白
- 独立只读审查：
  - 不改代码的 explorer 类工作
  - 不共享写入面的实现可行性判断
- 后续修复阶段里，若问题落在互不重叠的模块，可按模块拆 worktree 或拆子任务并行修

### 必须串行的块

- `批次 2A` 核心记录主链路：
  - 五维访谈
  - 单篇日志
  - 当天完整日志
  - 它们共享账号、`entryDate`、已保存维度日志和当天整合日志来源集合
- `批次 2B` AI 访谈效果验收：
  - 同一维度的访谈推进、partial 收束、标题治理、draft 质量门、fallback 表现，需要在同一轮上下文里连续观察
  - gratitude stitched、多维来源补齐、完整日志来源更新，本身依赖前一步已保存结果
- 状态恢复与工作区切换：
  - 刷新恢复、跨工作区切换、`saved / draft / stale` 投影，天然依赖前一个状态已经准确建立

### 当前建议执行顺序

1. `批次 2A-1`：五维访谈可启动 + choice 动作 + 单篇日志生成
2. `批次 2A-2`：单篇日志保存 + 完整日志入口/生成/保存
3. `批次 2B`：按维度做 AI 访谈效果验收
   - `joy`
   - `fulfillment`
   - `reflection`
   - `improvement`
   - `gratitude`
4. `批次 3`：状态恢复与工作区切换
5. `批次 4-7`：按已有顺序继续

### 建议执行组

| 执行组 | 包含内容 | 方式 | 原因 |
|---|---|---|---|
| `组 0` | 环境确认、测试账号表、验收日期表 | 串行 | 后续批次共享账号、`entryDate`、浏览器状态，先固定前置最省返工 |
| `组 1` | `A-01 ~ A-06` | 串行 | 共用认证链路和 cookie 状态 |
| `组 1.1` | `A-07` | 串行，单独隔离 | 会碰 localStorage、上次维度、恢复态，不能和主链混跑 |
| `组 1.2` | `A-08` | 串行，放最后 | 删号会销毁账号和数据，只能收尾执行 |
| `组 2` | `B-01 ~ B-06` | 串行 | 共用同一主账号、同一日期、同一来源日志集合 |
| `组 3` | `C-01 ~ C-06` | 串行 | 直接依赖组 2 产出的进行中 session、draft、saved、stale |
| `组 4` | `AI-01 ~ AI-05` | 按维度半串行 | 同一维度内部要连续观察；不同维度可拆账号或拆 `entryDate` 降低污染 |
| `组 5` | `D-01 ~ D-04` | 部分并行 | `D-04` 可前置，`D-01` 独立造错，`D-02/D-03` 依赖状态投影 |
| `组 6` | `E-01 ~ E-04` + `F-01 ~ F-05` | 冻结数据后分层执行 | Calendar 与 Analysis 共用同一套读模型和评分/日志投影 |
| `组 7` | `G-01 ~ G-04` | 可并行观察，建议后置 | 对主链依赖低，但要吃前面积累出的真实数据 |

### 页面分线约束

- 主链窗口：
  - `/login`
  - `/register`
  - `/interview`
  - 只跑 `A / B / C / AI`
- 回看窗口：
  - `/calendar`
  - `/analysis`
  - 只跑 `E / F`，以及依赖状态投影的 `D-02 / D-03`
- 补充窗口：
  - `/profile`
  - `/settings`
  - 只跑 `G`

执行约束：
- 每条线固定一个浏览器 profile 或独立无痕上下文。
- 不在同一个浏览器上下文里混跑“主账号主链路”和“多账号隔离/删号”。

### 为什么 AI 访谈效果必须单列

- 现有 launch 矩阵里的 `B-01` 到 `B-07` 更偏“链路是否可用”。
- 仓库里已经存在一套更细的访谈质量标准，分散在：
  - `docs/operator-runbook.md`
  - `docs/integration-guide.md`
  - `docs/theory/*-alignment.md`
- 如果不把这套标准显式收进 launch 验收，容易出现“能生成日志，但问法、收束、标题或质量门已经偏离理论目标”的假通过。

## 上线判断门槛

- `P0` 全部关闭后，才允许进入“是否上线”的判断。
- `P1` 必须有明确处理结论：修复完成或明确接受理由。
- `P2` 可以保留，但必须作为已知限制显式记录。
