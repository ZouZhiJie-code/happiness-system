# 月度分析聚合模块重构设计

## 1. 背景与原因

`src/features/analysis/aggregate-month.ts` 当前有 1227 行，同时承担月份日期构造、每日覆盖计算、幸福评分趋势、节奏统计、五维证据提取、维度洞察排序、跨维关系和总览文案等职责。它只有一个主要生产调用方，内部以确定性纯函数为主，并由 `analysis.service`、Analysis API 和前端壳层测试间接覆盖。

这使它成为适合首轮重构的范围：结构收益清晰，数据和文案契约稳定，测试基线充足，且与主工作区正在进行的流式日志改动没有文件重叠。当前完整测试基线为 914 项通过、1 项既有固定日期测试失败；与本模块直接相关的 16 项测试全部通过。

本轮目标是降低单文件职责密度，让日期覆盖、五维洞察和跨维总览可以独立理解与测试。用户可见文案、排序、日期语义、API 返回结构和数据库结构全部保持不变。

## 2. 方案比较与选择

### 方案 A：只拆出评分函数

改动最小，但主体文件仍会保留日期、五维和跨维叙事三类职责，长期维护收益有限。

### 方案 B：按数据流拆成内部模块（采用）

保留 `aggregate-month.ts` 作为稳定编排入口，新增共享日期工具、覆盖与评分、维度洞察、跨维总览四个内部模块。每个模块都有单向依赖，能够独立测试，也能继续通过原入口提供现有导出。

### 方案 C：建立通用聚合框架

可以进一步统一月分析和区间趋势，但当前两个聚合器的输出目的和时间窗口语义不同。此方案会扩大范围并引入提前抽象，本轮不采用。

## 3. 目标结构

- `aggregate-month.ts`
  - 保留 `aggregateAnalysisMonth`。
  - 从原路径继续导出 `buildAnalysisScoreTrend`。
  - 只负责按顺序编排各模块并组装最终记录。
- `month-aggregation-utils.ts`
  - 提供月份日期列表、日期倒序比较和一位小数舍入。
- `month-coverage.ts`
  - 负责每日覆盖、评分趋势、评分总览和月内节奏。
- `month-dimensions.ts`
  - 负责五维数量统计、证据提取、强度/连续性/动量和维度卡片。
- `month-insights.ts`
  - 负责维度排序、配对/跟随/缺口/评分关系以及月度总览。

依赖方向固定为：`aggregate-month` → 三个领域模块 → `month-aggregation-utils`。`month-insights` 只消费已经生成的维度卡和每日覆盖记录，不回调其他领域模块。

## 4. 数据流与稳定边界

输入继续由 `analysis.service.ts` 提供：月份、已保存维度日志、当天整合日志、幸福评分和今天日期。覆盖模块先生成完整月份天列表、每日状态、评分趋势和节奏；维度模块使用评分因子均值生成五维卡片；总览模块消费五维卡片与每日覆盖，生成关系和叙事；入口最后组装原有返回值。

必须保持以下契约：

1. `aggregateAnalysisMonth` 的参数和返回类型不变。
2. `buildAnalysisScoreTrend` 继续从原模块路径导入。
3. 无效月份继续抛出 `INVALID_MONTH`。
4. 评分只进入目标月份，平均值保留一位小数。
5. 未来月份不产生虚假的空白节奏。
6. 当天整合日志 stale、pending 与 saved 语义不变。
7. 五维排序、证据条数、标题和所有中文文案逐字保持。

## 5. 错误处理与测试

本层维持确定性纯函数，不新增捕获或错误包装。月份格式错误继续在日期构造边界抛出，仓库查询异常仍由 `analysis.service.ts` 转换为 `AnalysisQueryError`。

测试采用 TDD：先为新的内部模块路径写行为测试并确认因模块尚未存在而失败，再移动最小代码使其通过。随后运行原有 Analysis service、API、趋势聚合和 UI 测试，最后运行类型检查、lint 与完整测试。完整测试中的既有固定日期失败会单独记录，并通过重构前后相同失败证明本轮未引入该问题。
