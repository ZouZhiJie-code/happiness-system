后端 API 与认证安全评审

外部章节基线：07 backend api、08 auth security

对 Daily Light 的结论：这一组是当前仓库最接近“真实闭环”的部分。

判断：已完成到首版产品可用 本地稳定开发：强 小范围真实用户：可用 正式上线运营：还需补环境与监控

为什么判为已完成

  • 仓库里存在完整业务 API 面：`src/app/api/*` 下已落地 interview、draft、daily-journal、calendar、analysis、profile、auth、happiness-score 等 28 条路由。
  • API 不是薄壳摆设，对应服务层和 repository 真实存在，例如 `src/server/services/auth/*`、`src/server/services/calendar/*`、`src/server/services/analysis/*`、`src/server/services/daily-journal/*`。
  • 认证基础面完整：`User`、`AuthSession`、`passwordHash`、`tokenHash`、`httpOnly` `dl_session` cookie、页面守卫、请求鉴权、删号级联都在 Prisma schema 和服务层里真实落地。
  • 私有页面守卫不是前端假跳转，`/interview`、`/calendar`、`/analysis`、`/profile`、`/settings` 都通过 `requireAuthenticatedPage()` 做服务端拦截。
  • 账号删除不是 UI 假动作，`POST /api/auth/delete-account` 会要求当前密码，后端调用 `deleteUserById`，依赖 Prisma 级联删除用户相关数据。

还没做到哪一步

  • 这组能力完成的是“首版账号体系 + 私有产品闭环”，不是“成熟互联网账号体系”。没有找回密码、没有邮箱/手机号验证、没有明显的登录限流与设备管理。
  • 安全配置更多停留在应用层，正式上线前还应补充生产环境 cookie 策略复核、域名级 HTTPS、日志脱敏、异常监控与基础审计。
  • `/api/transcribe` 仍是 stub，说明“需要鉴权的业务 API”与“真正对用户开放的能力面”之间仍有一块故意关闭的区域。

强决策

  • 现在必须做：保持 auth/API 主链稳定，不要重构大框架；把生产环境变量、日志和部署入口补上，让这套已存在的业务 API 真能被外部环境承载。
  • 可以后置：密码找回、邮箱验证、多设备会话管理，这些在首版小范围试用前不是最高优先级。
  • 现阶段不建议：为了“看起来更安全”先大改认证模型。当前更缺的是发布与运维闭环，不是账号体系推倒重来。
