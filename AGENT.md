# Happiness-system-codex AGENT

## 1. 项目概述

本项目用于实现一个基于“幸福日志”理论的 AI 访谈产品。

当前第一版目标：
- 只实现 `开心（joy）` 维度
- 通过 `AI 访谈` 帮助用户回顾当天经历
- 支持 `文字输入` 与 `语音转文字`
- 生成一份 `可编辑、可确认` 的开心日志草稿

当前阶段不做：
- 充实、思考、改进、感谢等其他维度
- 多维度自由切换式开放聊天
- 实时全双工语音
- 强建议式教练输出

本仓库的核心原则不是“做一个会聊天的 AI”，而是“做一个质量稳定、可测试、可恢复的访谈系统”。

## 2. 开发总原则

后续所有开发必须遵循以下优先级：

1. 访谈质量稳定
2. 流程可控
3. 数据可追踪
4. 会话可恢复
5. 界面体验清晰

必须遵守：
- 不允许仅靠 prompt 控制访谈流程
- 流程控制必须由后端状态机完成
- 所有关键 AI 输出必须结构化，并经过服务端校验
- 所有关键状态必须持久化，避免刷新丢失
- 历史记忆必须可关闭，且只在用户开启时使用

如果 `AGENT.md` 与 `Tech_Design.md` 存在冲突，以：
1. 用户最新指令
2. `AGENT.md`
3. `Tech_Design.md`
为准。

## 3. 技术基线

当前默认技术选型：
- 前端：`Next.js 15`、`React 19`、`TypeScript`、`Tailwind CSS`
- 状态管理：`Zustand`
- 表单与校验：`React Hook Form`、`Zod`
- 后端：`Next.js Route Handlers` + service layer
- 数据库：`PostgreSQL` + `Prisma`

补充要求：
- 统一使用 `TypeScript`
- 默认开启严格类型约束
- 不引入与第一版目标无关的重型基础设施
- 第一版不引入独立向量数据库、消息队列、搜索引擎

## 4. 架构规范

### 4.1 分层规则

代码必须按职责分层：
- `src/app`：页面与 API 入口，不放复杂业务逻辑
- `src/components`：纯 UI 组件
- `src/features/joy-interview`：开心维度专属逻辑
- `src/server/services`：核心业务编排与 AI 能力
- `src/server/repositories`：数据库读写封装
- `src/lib` 与 `src/server/lib`：通用工具

禁止行为：
- 在页面组件中直接写 AI prompt 编排逻辑
- 在 React 组件中直接访问数据库
- 在 route handler 中堆积完整业务流程
- 让 prompt 自行决定是否结束访谈

### 4.2 AI 访谈链路

开心维度访谈必须拆成 4 个阶段：
1. `Transcribe`
2. `Extract`
3. `Orchestrate`
4. `Generate`

说明：
- `Transcribe` 负责语音转文字
- `Extract` 负责从用户回答中抽取结构化信息
- `Orchestrate` 负责决定下一步问题和会话状态
- `Generate` 负责生成最终日志草稿

任何单次模型调用都不应同时承担“理解、判断流程、生成草稿、决定结束”全部职责。

### 4.3 状态机要求

开心维度第一版必须使用显式状态机，至少包含：
- `collect_event`
- `probe_reason`
- `probe_pattern`
- `wrap_up`
- `finalize`

结束条件必须由代码判断，而不是由模型自由发挥。

第一版默认收束条件：
- 已拿到具体开心事件
- 已拿到为什么开心
- 已拿到开心类型或自我模式之一
- 或达到轮次上限

### 4.4 结构化输出要求

以下阶段必须返回结构化 JSON：
- `extract`
- `finalize`

要求：
- 服务端必须使用 `Zod` 做 schema 校验
- JSON 非法时允许自动重试一次
- 再次失败时返回可恢复错误
- 不允许因单次 AI 失败破坏整个会话

## 5. 数据与持久化规范

第一版至少要保证以下数据可追踪：
- 会话基础信息
- 每轮用户消息与 AI 消息
- 每轮抽取快照
- 最终日志草稿与正式日志
- AI 调用日志

必须遵守：
- 每轮对话后立即持久化消息
- 每轮抽取后立即持久化 snapshot
- 页面刷新后可恢复未完成 session
- 最终日志必须保留 `source` 与 `sessionId`

涉及用户历史记忆时：
- 默认关闭，依赖用户设置开启
- 单次会话中显式引用历史不超过 1 次
- 第一版最多向生成层提供 3 条相关历史记录

## 6. API 设计规范

API 设计遵循“薄入口、厚服务”原则。

要求：
- route handler 只负责鉴权、参数校验、调用 service、返回结果
- 请求入参与响应出参必须有明确 schema
- 错误返回必须稳定、可识别、可恢复
- 不把原始模型输出直接透传给前端

第一版核心接口以 `Tech_Design.md` 为准：
- `POST /api/interview/session/start`
- `POST /api/transcribe`
- `POST /api/interview/session/respond`
- `POST /api/interview/session/finalize`
- `GET /api/interview/session/:id`
- `PUT /api/joy-entry/:id`

## 7. 前端开发规范

前端目标是“低打扰、高完成率”，不是复杂交互炫技。

要求：
- 优先保证访谈流程清晰、输入反馈明确、错误恢复简单
- 录音失败时必须能无缝切换文本输入
- 用户必须能看见当前回答是否已保存
- 页面刷新后必须尽量恢复上下文
- 最终草稿必须允许编辑后再确认

不建议：
- 在第一版引入复杂动效
- 做开放式聊天 UI
- 把重点放在装饰性组件而忽略可用性

## 8. 测试要求

没有测试设计的代码，不算完成。

### 8.1 最低测试覆盖

第一版至少包含以下测试：
- `单元测试`
  - 访谈状态机流转
  - extract/finalize schema 校验
  - 收束条件判断
  - 历史记忆筛选规则
- `集成测试`
  - session start/respond/finalize 主流程
  - AI 返回非法 JSON 时的重试与降级
  - 页面刷新后的 session 恢复
- `手工验证`
  - 文字输入完整跑通一次开心访谈
  - 语音转文字失败后切文本继续
  - 生成草稿后编辑并保存

### 8.2 测试标准

测试必须覆盖以下风险：
- 用户回答很短
- 用户回答很散，没有直接说出开心原因
- 用户连续两轮表达重复
- 模型返回缺字段
- 模型返回格式错误
- 会话中途刷新
- 用户主动结束访谈

### 8.3 提交前检查

提交前至少确认：
- 类型检查通过
- lint 通过
- 关键单元测试通过
- 主流程可以本地手工跑通

如果某项未完成，必须明确记录原因，不能默认忽略。

## 9. 代码风格

### 9.1 通用要求

- 使用 `TypeScript`
- 优先写小函数、纯函数、明确返回值
- 不滥用 `any`
- 非必要不写注释，必要注释只解释“为什么”
- 命名必须表达业务含义，避免 `data`、`info`、`temp`

### 9.2 命名约定

- React 组件：`PascalCase`
- hooks：`useXxx`
- 工具函数：`camelCase`
- 文件名：优先 `kebab-case`
- schema 文件：包含 `.schema`
- server-only 逻辑不得混入 client 组件

### 9.3 类型与校验

- 能用联合类型表达状态时，不用松散字符串
- 接口入参与 AI 输出都先过 schema，再进入业务逻辑
- 前后端共享的数据结构，优先抽到独立 schema 或 type 文件

### 9.4 错误处理

- 不吞错
- 错误必须包含可定位的信息
- 面向用户的错误文案保持稳定、简洁、可理解
- 面向日志的错误信息应保留 stage、sessionId、原因

## 10. 可观察性与日志

第一版必须预留可观察性能力，至少记录：
- 转写耗时
- 抽取耗时
- 生成耗时
- 当前缺失槽位
- 本轮状态机状态
- 结束原因
- AI 调用是否成功

日志要求：
- 区分开发日志与业务日志
- 不在日志中泄露不必要的用户隐私
- 能用结构化日志就不用散乱字符串

## 11. 注意事项

### 11.1 产品边界

- 第一版的目标是验证“开心访谈是否能稳定帮助用户梳理感受”
- 不要提前设计五维统一大系统
- 不要为了扩展性牺牲第一版可落地性

### 11.2 AI 边界

- AI 在第一版中主要负责提问、抽取、整理
- AI 不承担心理咨询、诊断、强建议职责
- AI 可以做轻提示，但不应替用户下结论

### 11.3 用户体验边界

- 访谈感受应像“被认真引导回忆”，不是“被系统盘问”
- 问题数量要克制
- 同类追问不能机械重复
- 最终结果要让用户觉得“被理解并被整理清楚了”

### 11.4 工程边界

- 不使用旧仓库 `幸福系统设计/happiness-system`
- 所有新开发均在 `Happiness-system-codex` 中完成
- 未经明确需要，不提前接入额外服务

## 12. 完成定义

一个功能只有在同时满足以下条件时，才算完成：
- 符合当前产品范围
- 符合本文件约束
- 代码分层清晰
- 关键状态可持久化
- 关键路径有测试或可验证方案
- 出错时可以恢复，而不是直接中断
- 用户可以实际操作并完成一次闭环

后续开发默认遵循本文件，不再重复说明。
